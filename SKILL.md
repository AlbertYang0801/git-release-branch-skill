# Git Release Branch

将部署单中的服务和提测分支转换为受确认门保护、可审计的 Git 发布操作。所有服务独立执行和记录；整体状态只能是 `成功 | 部分失败 | 失败 | 中止`。

## 不可跳过的原则

- 任何改变本地分支、工作区或远端状态的命令执行前，先展示计划及实际命令并获得用户明确确认。
- 检查类命令可在确认前执行；执行前先 `git fetch --prune origin` 和 `git fetch --tags origin`。
- 不自动解决冲突，不删除分支，不强推，不覆盖已有 release 分支或 tag。
- `master` 含 release 未包含的提交、release/tag 已存在、merge 冲突或 push 失败时，停止该服务的后续破坏性操作。
- 只记录实际执行过的命令；凭据、令牌和敏感参数统一替换为 `***`。
- 无论成功、失败、风险拦截、用户拒绝确认或中途取消，都必须写运行结果。

## 持久化位置

首次使用且尚未确定服务路径表位置时，必须询问用户确认保存位置。推荐默认路径为 `%USERPROFILE%\.codex\git-release-branch\services.md`（展开为当前操作系统用户的主目录），同时允许用户提供任意可写的自定义路径。不得静默采用推荐路径，也不得硬编码某个用户名。确认后在本次及后续流程中复用该位置：

```markdown
# Git Release Branch Services

| 服务名 | 本地仓库地址 |
|---|---|
| whds-app-manage | D:\YzhlProjects\whds3-jdk8\whds-app-manage |
```

若用户选择的位置不存在，则在展示将创建的目录和文件后再次确认，再创建文件。服务未登记时询问本地仓库地址，确认路径存在且是 Git 仓库后再写入。不得猜测路径。

运行结果写入本 skill 根目录的 `runs/`：

- 上预发：`runs/YYYYMMDD-HHMMSS-prerelease.md`
- 生产后打 tag：`runs/YYYYMMDD-HHMMSS-tag.md`

若安装位置只读，先询问用户一个可写的结果目录，并在最终回复中明确实际文件路径。

## 读取发布输入

支持飞书文档链接、粘贴文本、本地文件和手动输入。飞书链接优先使用可用的飞书文档工具或 `lark-cli`；读取失败时请用户粘贴内容。每个发布项必须得到服务名和来源分支，缺项时询问，不自行推断。

## 版本推导

仅识别 `Vx.y.z` tag，并基于每个仓库最新语义版本计算：

| 类型 | 规则 | 示例 |
|---|---|---|
| 大版本 | `minor + 1`，`patch = 0` | `V3.7.1` → `V3.8.0` |
| 小版本 | `patch + 1` | `V3.7.1` → `V3.7.2` |

release 名为 `release/Vx.y.z_YYYYMMDD`。日期取当前机器当天，第二轮确认前允许用户覆盖。

## 场景一：上预发

1. 读取输入，提取服务名和来源分支，从 `services.md` 补齐本地路径。
2. 展示第一轮确认表：`服务名 | 发布分支 | 服务本地地址`。用户未确认则写“中止”记录并结束。
3. 对每个仓库检查路径、Git 仓库、远端来源分支和最新 tag。检查失败只停止对应服务。
4. 询问大版本或小版本，计算 release 名。
5. 检查本地和远端 release 是否存在；存在时不得覆盖，让用户选择复用现有分支、调整版本/日期或跳过。
6. 展示第二轮确认表：`服务名 | 来源分支 | 最新 tag | 版本类型 | release 分支名`，并列出将实际执行的命令。
7. 用户确认后逐服务执行等价操作：切到来源分支、拉取最新、从该提交新建 release、推送 `origin`。命令以仓库实际分支形态为准，展示内容必须与实际执行一致。
8. 汇总每个服务并写 `prerelease` 结果文件。

## 场景二：生产后打 tag

1. 从用户输入或既有 prerelease 记录取得服务名和 release 分支，并从 `services.md` 补齐路径。
2. fetch 后确认远端 release 与 master 存在；从 `release/V3.8.0_20260709` 推导 tag `V3.8.0`，无法解析时要求用户确认 tag。
3. 分别检查 `origin/release..origin/master` 与 `origin/master..origin/release`，展示：`服务名 | release 分支 | master 多出提交数 | release 待合入提交数 | 风险判断`。
4. master 多出提交时展示提交摘要，标记风险拦截并停止该服务；不得以用户口头确认绕过重新检查。
5. 差异通过后检查本地和远端 tag。tag 已存在时停止，询问复用现有 tag 或中止，禁止覆盖。
6. 展示合并、push master、创建 tag、push tag 的实际命令，获得第二个明确确认门。
7. 确认后依次执行；merge 冲突时停止且不自动处理，push master 失败时不得创建 tag。只有 master push 成功且 tag 仍不存在时，才基于当前 master 创建并推送 tag。
8. 汇总每个服务并写 `tag` 结果文件。

多服务中只要同时出现成功和失败/中止，整体必须写“部分失败”；全部未执行且由用户取消或风险门拦截时写“中止”。最终回复列出成功服务、失败或中止服务、原因及结果文件路径。

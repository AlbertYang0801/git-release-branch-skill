# Git Release Branch

将部署单中的服务和提测分支转换为受确认门保护、可审计的 Git 发布操作。所有服务独立执行和记录；整体状态只能是 `成功 | 部分失败 | 失败 | 中止`。

## 不可跳过的原则

- 任何改变本地分支、工作区或远端状态的命令执行前，先展示计划及实际命令并获得用户明确确认。
- 解析完全部服务及仓库路径后，第一项 Git 检查必须是全量工作区检查。只有所有服务的工作区都干净，才允许执行 `git fetch --prune origin`、`git fetch --tags origin` 及后续步骤。
- 不自动解决冲突，不删除分支，不强推，不覆盖已有 release 分支或 tag。
- `master` 含 release 未包含的提交、release/tag 已存在、merge 冲突或 push 失败时，停止该服务的后续破坏性操作。
- 每次运行必须取得并确认本次上线需求名称。未提供时必须询问；未确认前不得执行创建 release、合并、push 或创建 tag 等变更操作。
- 只记录实际执行过的命令；凭据、令牌和敏感参数统一替换为 `***`。
- 总体状态与阶段状态分开记录。任一后续阶段失败时，不能用总体“失败”掩盖此前已经成功产生的本地或远端变更。
- 取得并确认有效需求名称后，无论成功、失败、风险拦截、用户拒绝确认或中途取消，都必须写运行结果。取得有效需求名称前用户取消或不再回复时，不生成运行记录。

## 持久化位置

首次使用且尚未确定服务路径表位置时，必须询问用户确认保存位置。推荐默认路径为 `%USERPROFILE%\.codex\git-release-branch\services.md`（展开为当前操作系统用户的主目录），同时允许用户提供任意可写的自定义路径。不得静默采用推荐路径，也不得硬编码某个用户名。确认后在本次及后续流程中复用该位置：

```markdown
# Git Release Branch Services

| 服务名 | 本地仓库地址 |
|---|---|
| whds-app-manage | D:\YzhlProjects\whds3-jdk8\whds-app-manage |
```

若用户选择的位置不存在，则在展示将创建的目录和文件后再次确认，再创建文件。服务未登记时询问本地仓库地址，确认路径存在且是 Git 仓库后再写入。不得猜测路径。

运行结果写入本 skill 根目录的 `runs/`。最终状态确定后一次性创建文件，不创建“执行中”临时文件：

```text
runs/YYYY-MM-DD-<清洗后的需求名称>-<场景>-<状态>.md
```

- 场景只能是 `release | tag`。
- 状态只能是 `成功 | 部分失败 | 失败 | 中止`。
- 示例：`runs/2026-07-17-用户权限改造V2-tag-成功.md`、`runs/2026-07-17-用户权限改造V2-release-失败.md`。
- 清洗需求名称时删除 Windows 文件名非法字符 `\ / : * ? " < > |`、中文全角冒号 `：`，并删除所有空白字符，包括空格、Tab 和换行；正文仍保留原始需求名称。
- 清洗后名称为空时要求用户重新提供，不得用服务名、版本号或日期替代。
- 只有用户确认清洗后非空的需求名称，才视为运行正式开始；在此之前取消不生成运行记录。
- 同名文件已存在时不得覆盖，在扩展名前追加从 `-2` 开始的最小可用递增序号，例如 `2026-07-17-用户权限改造V2-tag-成功-2.md`。
- 已有旧格式记录保持不变，不批量重命名。

若安装位置只读，先询问用户一个可写的结果目录，并在最终回复中明确实际文件路径。

## 读取发布输入

支持飞书文档链接、粘贴文本、本地文件和手动输入。飞书链接优先使用可用的飞书文档工具或 `lark-cli`；读取失败时请用户粘贴内容。每次运行必须得到本次上线需求名称；每个发布项必须得到服务名和来源分支，缺项时询问，不自行推断。

## 版本推导

仅识别 `Vx.y.z` tag，并基于每个仓库最新语义版本计算：

| 类型 | 规则 | 示例 |
|---|---|---|
| 大版本 | `minor + 1`，`patch = 0` | `V3.7.1` → `V3.8.0` |
| 小版本 | `patch + 1` | `V3.7.1` → `V3.7.2` |

release 名为 `release/Vx.y.z_YYYYMMDD`。日期取当前机器当天，第二轮确认前允许用户覆盖。

## 全量工作区前置检查

取得全部服务及本地仓库路径后，对每个服务执行以下只读检查，必须检查完所有服务再汇总，不得遇到第一个异常就提前结束：

```text
git -C <仓库地址> status --porcelain=v1 --untracked-files=all
```

- 空输出表示工作区、暂存区和未跟踪文件均为空。
- 非空输出必须按服务汇总暂存、未暂存和未跟踪内容，展示：`服务名 | 仓库地址 | 工作区状态 | 待处理内容`。
- 任一服务非空时暂停整个发布流程，提示用户处理；不得对任何服务执行 fetch、切换分支、pull、创建 release、merge、push 或创建 tag。
- 用户处理后要求重新检查全部服务，不能只复查此前异常的服务。
- 全部服务检查通过后，才进入 fetch 和远端状态检查。
- 若用户选择取消：已确认有效需求名称时写“中止”记录；尚未确认有效需求名称时不生成记录。

## 场景一：上预发

1. 读取输入，取得本次上线需求名称，提取服务名和来源分支，从 `services.md` 补齐本地路径。需求名称缺失或清洗后为空时先要求用户补充。
2. 按“全量工作区前置检查”检查全部服务；任一服务存在未提交内容时暂停整个流程，全部干净后才继续。
3. 展示第一轮确认表：`需求名称 | 服务名 | 发布分支 | 服务本地地址`。用户未确认则写“中止”记录并结束。
4. fetch 后对每个仓库检查路径、Git 仓库、远端来源分支和最新 tag。检查失败只停止对应服务。
5. 询问大版本或小版本，计算 release 名。
6. 检查本地和远端 release 是否存在；存在时不得覆盖，让用户选择复用现有分支、调整版本/日期或跳过。
7. 展示第二轮确认表：`服务名 | 来源分支 | 最新 tag | 版本类型 | release 分支名`，并列出将实际执行的命令。
8. 用户确认后逐服务执行等价操作：切到来源分支、拉取最新、从该提交新建 release、推送 `origin`。命令以仓库实际分支形态为准，展示内容必须与实际执行一致。
9. 汇总每个服务的 release 创建、release 推送阶段状态，并按 `release` 场景及最终状态写结果文件。

## 场景二：生产后打 tag

1. 从用户输入或既有 release 记录取得本次上线需求名称、服务名和 release 分支，并从 `services.md` 补齐路径。需求名称缺失或清洗后为空时先要求用户补充。
2. 按“全量工作区前置检查”检查全部服务；任一服务存在未提交内容时暂停整个流程，全部干净后才继续。
3. fetch 后确认远端 release 与 master 存在；从 `release/V3.8.0_20260709` 推导 tag `V3.8.0`，无法解析时要求用户确认 tag。
4. 分别检查 `origin/release..origin/master` 与 `origin/master..origin/release`，展示：`服务名 | release 分支 | master 多出提交数 | release 待合入提交数 | 风险判断`。
5. master 多出提交时展示提交摘要，标记风险拦截并停止该服务；不得以用户口头确认绕过重新检查。
6. 差异通过后检查本地和远端 tag。tag 已存在时停止，询问复用现有 tag 或中止，禁止覆盖。
7. 在任何合并、push 或创建 tag 前，展示确认表：`需求名称 | 服务名 | release 分支 | tag | 风险判断`，同时列出合并、push master、创建 tag、push tag 的实际命令，获得第二个明确确认门。用户必须明确确认本次上线需求名称；未确认则写“中止”记录并结束。
8. 确认后依次执行；merge 冲突时停止且不自动处理，push master 失败时不得创建 tag。只有 master push 成功且 tag 仍不存在时，才基于当前 master 创建并推送 tag。
9. 若 master 推送成功但 tag 创建或推送失败，服务总体状态记为“失败”，同时明确记录 `master推送: 成功`、tag 失败阶段及 `需人工处理: 是`。后续动作只能针对 tag；不得重新合并或重复推送 master，重试前必须重新检查远端 master 与 tag。
10. 汇总每个服务的 master 合并、master 推送、tag 创建、tag 推送阶段状态并写 `tag` 结果文件。

## 运行结果模板

```markdown
---
场景: release | tag
状态: 成功 | 部分失败 | 失败 | 中止
需求名称: 用户提供的原始需求名称
开始时间: YYYY-MM-DD HH:mm:ss
结束时间: YYYY-MM-DD HH:mm:ss
操作者: 用户提供或未提供
输入来源: 飞书文档 | 粘贴文本 | 本地文件 | 手动输入
---

## 执行摘要

| 服务名 | 来源分支 | release 分支 | tag | 状态 | 说明 |
|---|---|---|---|---|---|

## 阶段状态

| 服务名 | release创建 | release推送 | master合并 | master推送 | tag创建 | tag推送 | 需人工处理 |
|---|---|---|---|---|---|---|---|

阶段字段只能填写 `成功 | 失败 | 未执行 | 不适用`；`需人工处理` 只能填写 `是 | 否`。上预发场景的 master/tag 字段填写“不适用”，生产后打 tag 场景的 release 创建/推送字段填写“不适用”。只要某一步已经成功改变本地或远端状态而后续步骤失败，必须填写 `需人工处理: 是`，并在“说明”和“失败原因与后续动作”中写明已经完成的阶段、失败阶段以及禁止重复执行的步骤。

## 已执行命令

```text
仅记录已实际执行的命令；敏感信息使用 *** 遮蔽。
```

## 失败原因与后续动作

- 无失败时写“无”。
- 失败或中止时逐项给出原因和人工下一步。
```

多服务中只要同时出现成功和失败/中止，整体必须写“部分失败”；全部未执行且由用户取消或风险门拦截时写“中止”。单服务后续阶段失败时整体仍写“失败”，由阶段状态表达已经发生的副作用。最终回复必须列出成功服务、失败或中止服务、各阶段状态、是否需人工处理、原因及结果文件路径。

# git-release-branch

当前版本：`0.2.3`

用于团队 Git 发布流程的 Codex skill，覆盖：

- 从部署单中的提测分支创建并推送 `release/Vx.y.z_YYYYMMDD`；
- 生产发布后检查 release/master 双向差异，安全合并 master 并创建、推送版本 tag；
- 在创建 release 或生产后打 tag 前确认本次上线需求名称；
- 所有成功、失败和中止路径写入可审计的 Markdown 运行记录。

## 安装

复制整个 `git-release-branch-skill` 目录到个人 skills 目录，保持 `SKILL.md` 与 `runs/` 的相对位置不变。skill 内容不依赖旁边的需求跟踪目录。

## 使用示例

- “根据这个部署单上预发并创建 release 分支。”
- “生产发完了，检查 release 与 master 后打 tag。”

首次使用时会先询问服务路径表的保存位置，推荐 `%USERPROFILE%\.codex\git-release-branch\services.md`，也可选择其他可写路径；确认后，再在首次遇到服务时询问本地仓库路径并维护该文件。每次运行还必须提供并确认本次上线需求名称；取得有效需求名称前取消不生成运行记录。解析完全部服务后，首先检查所有仓库的暂存、未暂存和未跟踪内容；任一仓库存在未提交内容时暂停整个流程，全部处理干净并重新全量检查后才继续。任何修改分支、合并、push 或 tag 操作都需要用户确认。

## 运行记录

运行结束后按最终状态一次性写入：

```text
runs/YYYY-MM-DD-<需求名称>-<release|tag>-<成功|部分失败|失败|中止>.md
```

需求名称会删除 `\ / : * ? " < > |`、中文全角冒号 `：` 和所有空白字符；正文保留原始名称。清洗后为空时必须重新提供。同名记录不会覆盖，依次追加 `-2`、`-3`。例如：

- `runs/2026-07-17-用户权限改造V2-tag-成功.md`
- `runs/2026-07-17-用户权限改造V2-release-失败.md`

运行记录除总体状态外，还会记录 `release创建`、`release推送`、`master合并`、`master推送`、`tag创建`、`tag推送` 和 `需人工处理`。例如 master 已推送但 tag 推送失败时，总体状态为“失败”，阶段状态明确标记 `master推送: 成功`、`tag推送: 失败`、`需人工处理: 是`，避免重复合并或重复推送 master。

## 文件

- `SKILL.md`：触发条件、发布流程、安全门和运行记录格式。
- `runs/`：每次执行的原始结果；仓库中的两份旧格式记录是早期文档验收样例，不做重命名。
- `tests/`：Skill 结构契约、运行记录命名和场景定义回归测试。

## 测试

在 PowerShell 中运行：

```powershell
.\tests\run-tests.ps1
```

测试只读取 `SKILL.md`、`README.md` 和场景数据，并在内存中验证命名规则；不会访问业务 Git 仓库或远端。

## 版本记录

### 0.2.3

- 增加可重复执行的 Skill 契约、运行记录命名和场景定义测试。
- 将中文全角冒号纳入需求名称清洗规则。

### 0.2.2

- 增加 Release、master 和 Tag 的强制阶段状态记录。
- master 推送成功但 Tag 失败时，明确标记已完成副作用和人工处理要求。

### 0.2.1

- 增加所有服务的全量工作区前置检查。
- 任一仓库存在未提交或未跟踪内容时暂停整个发布流程，处理后重新检查全部服务。

### 0.2.0

- 创建 Release 或生产后打 Tag 前确认本次上线需求名称。
- 运行记录文件名包含日期、需求名称、场景和最终状态。
- 清洗文件名非法字符和空白，并避免覆盖同名记录。
